1. Git Rebasing

  • Git Rebasing is a process used in version control with Git to integrate changes from one branch into another.
  • It allows user to move or apply a series of commits from one branch onto another branch, typically resulting in a more linear and cleaner commit history.
  • Rebasing is useful for maintaining a clear history and resolving conflicts between branches.

How to Rebase ?

  1. Start with a base branch (e.g., master or main).
  2. Create a feature branch.
  3. Make your changes in the feature branch.
  4. Move the changes from the feature branch onto the latest commit in the base branch by using below command.

    5.     git rebase [base_branch]

Result : The commit history looks as if the changes in the feature branch were made directly on top of the base branch, creating a more straightforward timeline.


2. Git Hooks

  • Git hooks are custom scripts or programs that user can set up to be automatically executed at specific points in the Git version control process.
  • These scripts allow user to automate tasks, enforce coding standards, perform pre-commit checks, send notifications, and more.
  • There are two types of hooks present in Git.
    1. Client-Side Hooks
    2. Server-Side Hooks

Client-Side Hooks
  • pre commit, prepare-commit-msg, commit-msg, post-commit, post-checkout, pre-rebase.
Server-Side Hooks
  • pre-receive, update, post-receive.

How to use Git Hooks ?

  1. Navigate to Your Git Repository.
  2. Locate the Hooks Directory: cd /path/to/your/repo/.git/hooks
  3. Choose or Create a Hook Script.
  4. Write Your Hook Script: To create a custom pre-commit hook, you can create a file named pre-commit in the hooks directory
  5. Customize Your Script.
  6. Testing (Optional).
  7. Commit Your Changes:
    8.     git add .git/hooks/pre-commit
    git commit -m "Add custom pre-commit hook"
  8. Finally you can use the Hook.

3. Cherry Picking

  • Cherry Picking means selectively using data that supports a specific viewpoint while ignoring contradictory evidence.
  • Leads to presenting an incomplete or biased view of a subject, leading to misunderstandings.
  • Considered a logical fallacy, undermining the credibility of an argument or individual.
  • Contributes to the spread of misinformation and misinterpretation of issues.
  • Should be avoided by conducting a comprehensive analysis that includes all relevant data and evidence.

How to Avoid Cherry Picking?

  • Conduct thorough research: Gather a wide range of sources and data to get a holistic understanding of the topic.
  • Acknowledge opposing viewpoints: Consider and address conflicting evidence or perspectives to provide a balanced analysis.
  • Use systematic approaches: Employ standardized methodologies or protocols for data collection and analysis to minimize biases.
  • Engage peer review: Encourage feedback from peers or experts in the field to ensure the objectivity and integrity of your work.
  • Be transparent: Clearly communicate the methodology and sources used, along with any limitations or constraints, to maintain transparency and trustworthiness.

4. Git Stash

  • 'git stash' is a Git command used to temporarily store changes that are not ready to be committed to the repository.
  • It allows you to store modified tracked files and staged changes without committing them, enabling you to work on a different branch or address an urgent issue.
  • Stashed changes can later be reapplied or removed using the 'git stash apply' or 'git stash drop' commands, respectively.
  • Multiple stashes can be created, and you can apply changes from a specific stash by referencing its index.
  • It's a helpful tool for managing unfinished changes, allowing developers to switch contexts or deal with urgent matters without the need for a permanent commit.

How to use git stash?

  1. Stash your changes: Use git stash to store your modifications. This command will stash both staged and unstaged changes. 
            git stash
  2. View your stashed changes: You can use 'git stash list' to view the list of stashes you've created 
            git stash list
  3. Apply stashed changes: To reapply your stashed changes, you can use 'git stash apply'. This will reapply the most recent stash. 
            git stash apply
  4. Apply a specific stash: If you have multiple stashes, you can specify the stash by index using 'git stash apply' followed by the stash reference (e.g., 'git stash apply stash@{2}'). 
            git stash apply <-stash@{index}->
  5. Remove stashed changes: You can use 'git stash drop' to remove the most recent stash, or specify a specific stash to drop using its index. 
            git stash drop
        git stash drop <-stash@{index}->
